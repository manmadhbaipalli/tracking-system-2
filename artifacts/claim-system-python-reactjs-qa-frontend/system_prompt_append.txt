#### Step 1: Verify and Install Test Dependencies

#### Step 2: Set Up MSW (Mock Service Worker) for API Mocking

Create `src/test/mocks/handlers.ts` with mock API handlers for ALL endpoints the application consumes:

```typescript
import { http, HttpResponse } from 'msw';

export const handlers = [
  http.post('/api/auth/login', () => {
    return HttpResponse.json({
      token: 'mock-jwt-token',
      user: { id: 1, email: 'test@example.com', name: 'Test User' },
    });
  }),

  http.get('/api/users', () => {
    return HttpResponse.json({
      content: [
        { id: 1, email: 'user1@example.com', name: 'User One' },
        { id: 2, email: 'user2@example.com', name: 'User Two' },
      ],
      totalPages: 1,
      totalElements: 2,
    });
  }),

  // Add handlers for ALL API endpoints used by the application
];
```

Create `src/test/mocks/server.ts`:
```typescript
import { setupServer } from 'msw/node';
import { handlers } from './handlers';

export const server = setupServer(...handlers);
```

Update `src/test/setup.ts` to start/stop the MSW server:
```typescript
import '@testing-library/jest-dom';
import { server } from './mocks/server';

beforeAll(() => server.listen({ onUnhandledRequest: 'error' }));
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

**CRITICAL:** Use `onUnhandledRequest: 'error'` to catch any API calls that don't have mock handlers -- this prevents tests from silently making real HTTP requests.

#### Step 3: Write Component Unit Tests

Write tests for EVERY reusable component in `src/components/`. For each component, create a test file: `src/components/__tests__/ComponentName.test.tsx`.

Test coverage per component:
- **Renders correctly** -- component mounts without errors and displays expected content
- **Props handling** -- component renders differently based on props
- **User interactions** -- clicks, form inputs, toggles produce expected results
- **Conditional rendering** -- elements appear/disappear based on state or props
- **Accessibility** -- interactive elements are accessible (proper roles, labels)

Example component test:
```typescript
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { Header } from '../Header';
import { AuthProvider } from '../../context/AuthContext';
import { BrowserRouter } from 'react-router-dom';

const renderWithProviders = (ui: React.ReactElement) => {
  return render(
    <BrowserRouter>
      <AuthProvider>{ui}</AuthProvider>
    </BrowserRouter>
  );
};

describe('Header', () => {
  it('renders the application title', () => {
    renderWithProviders(<Header />);
    expect(screen.getByRole('banner')).toBeInTheDocument();
  });

  it('shows login link when user is not authenticated', () => {
    renderWithProviders(<Header />);
    expect(screen.getByRole('link', { name: /login/i })).toBeInTheDocument();
  });

  it('shows logout button when user is authenticated', () => {
    // Set up authenticated state
    localStorage.setItem('token', 'mock-token');
    renderWithProviders(<Header />);
    expect(screen.getByRole('button', { name: /logout/i })).toBeInTheDocument();
  });
});
```

**RULE:** Always wrap components in necessary providers (BrowserRouter, AuthProvider, etc.) in tests.

#### Step 6: Write Hook Tests

```typescript
import { renderHook, act } from '@testing-library/react';
import { useForm } from '../useForm';

describe('useForm', () => {

    const { result } = renderHook(() =>
```

## Reference: Best Practices

## React Testing Best Practices

### Test User Behavior, Not Implementation

- **DO** test what the user sees and interacts with: rendered text, button clicks, form submissions, navigation
- **DO NOT** test internal component state, lifecycle methods, or implementation details
- **DO NOT** test that `useState` was called with a specific value
- **DO** test that clicking "Add to Cart" shows "Item added" on the screen
- If you refactor a component's internals without changing its behavior, tests should still pass
- The guiding principle: "The more your tests resemble the way your software is used, the more confidence they can give you" (Kent C. Dodds)

### Use screen Queries (Priority Order)

Queries should reflect how users and assistive technologies find elements. Use this priority order:

1. **`getByRole`** -- best for accessibility, works like a screen reader
   ```typescript
   screen.getByRole('button', { name: /submit/i })
   screen.getByRole('heading', { name: /dashboard/i })
   screen.getByRole('textbox', { name: /email/i })
   screen.getByRole('navigation')
   ```

2. **`getByLabelText`** -- best for form inputs
   ```typescript
   screen.getByLabelText(/email address/i)
   screen.getByLabelText(/password/i)
   ```

3. **`getByPlaceholderText`** -- when no label exists (prefer adding a label)
   ```typescript
   screen.getByPlaceholderText(/search.../i)
   ```

4. **`getByText`** -- for non-interactive elements displaying text
   ```typescript
   screen.getByText(/welcome back/i)
   screen.getByText(/no results found/i)
   ```

5. **`getByDisplayValue`** -- for inputs with current values
   ```typescript
   screen.getByDisplayValue('john@example.com')
   ```

6. **`getByTestId`** -- LAST RESORT ONLY. Use only when no semantic query works.
   ```typescript
   screen.getByTestId('custom-chart-component')
   ```

**NEVER query by CSS class name or component internal structure.**

### Avoid Testing Internal State

- [ ] Never import `useState` or `useReducer` from a component to inspect state
- [ ] Never access `component.state` or `component.instance()`
- [ ] Never test that a specific function was called internally (unless it's a prop callback)
- [ ] Test the OUTCOME: "after clicking delete, the item disappears from the list"
- [ ] Test the SIDE EFFECT: "after submitting the form, a success message appears"
- [ ] If a test is hard to write without testing internals, the component may need refactoring

### MSW for API Mocking

- [ ] Use Mock Service Worker (MSW) for all API mocking -- it intercepts at the network level
- [ ] Define default handlers in `src/test/mocks/handlers.ts` for happy-path responses
- [ ] Override handlers per-test for error scenarios using `server.use()`
- [ ] Set `onUnhandledRequest: 'error'` to catch unmocked API calls
- [ ] Mock ALL endpoints the application consumes -- leave nothing unmocked
- [ ] Test both success AND failure responses for every API call
- [ ] Handlers should return realistic data shapes matching the real API contract

Example pattern for testing errors:
```typescript
import { server } from '../../test/mocks/server';
import { http, HttpResponse } from 'msw';

it('shows error when fetch fails', async () => {
  server.use(
    http.get('/api/items', () => {
      return new HttpResponse(null, { status: 500 });
    })
  );

  render(<ItemList />);
  expect(await screen.findByText(/something went wrong/i)).toBeInTheDocument();
});
```

### Test Accessibility with jest-axe

- [ ] Install `jest-axe` and extend matchers with `toHaveNoViolations`
- [ ] Run `axe()` on every page component's rendered output
- [ ] Fix violations instead of suppressing them
- [ ] Common violations to watch for:
  - Missing `<label>` on form inputs (use `htmlFor` or wrap in `<label>`)
  - Missing `alt` attribute on `<img>` tags
  - Buttons without accessible text (use `aria-label` for icon-only buttons)
  - Heading hierarchy violations (don't skip from `<h1>` to `<h3>`)
  - Missing landmark regions (`<main>`, `<nav>`, `<header>`)
  - Low color contrast ratios
- [ ] Accessibility testing is not optional -- it catches real usability issues

### Snapshot Tests -- Use Sparingly

- [ ] Snapshot tests are NOT a substitute for behavioral tests
- [ ] Use snapshots ONLY for stable, rarely-changing components (icons, static layouts)
- [ ] Never use snapshots for components with dynamic content (dates, IDs, random values)
- [ ] When a snapshot breaks, READ the diff carefully -- do not blindly update
- [ ] Prefer explicit assertions (`expect(screen.getByText(...)).toBeInTheDocument()`) over snapshots
- [ ] If you find yourself updating snapshots frequently, replace them with behavioral tests

### Test Error States and Loading States

- [ ] EVERY component that fetches data must have tests for:
  - **Loading state:** verify spinner/skeleton is shown during fetch
  - **Success state:** verify data is rendered correctly
  - **Error state:** verify error message is shown when API fails
  - **Empty state:** verify empty message when API returns empty array
- [ ] Use `findByText` (async) for content that appears after loading
- [ ] Use `queryByText` to verify content is NOT present (returns null instead of throwing)
- [ ] Test loading -> success transition:
  ```typescript
  render(<UserList />);
  expect(screen.getByText(/loading/i)).toBeInTheDocument();
  expect(await screen.findByText('User One')).toBeInTheDocument();
  expect(screen.queryByText(/loading/i)).not.toBeInTheDocument();
  ```

### Coverage Targets

- [ ] **Overall statement coverage: >= 80%** (enforced by the QA verdict)
- [ ] **Components: >= 85%** -- every component renders and responds to interactions
- [ ] **Pages: >= 80%** -- every page loads data and handles all states
- [ ] **Hooks: >= 90%** -- hooks are pure logic and should be fully testable
- [ ] **Services: >= 90%** -- service functions are straightforward to test with MSW
- [ ] **Utils: >= 95%** -- utility functions are pure and trivially testable
- [ ] Do NOT waste effort testing:
  - Type definitions (`src/types/`)
  - Constants and enums (unless they have logic)
  - Auto-generated code
  - CSS/Tailwind classes

### Async Testing Patterns

- [ ] Use `findBy*` queries for elements that appear asynchronously (they retry with a timeout)
- [ ] Use `waitFor()` when you need to assert on something after an async operation
- [ ] Use `waitForElementToBeRemoved()` when testing that an element disappears
- [ ] Never use `setTimeout` or arbitrary delays in tests -- always use Testing Library's async utilities
- [ ] Set up `userEvent` properly: `const user = userEvent.setup()` before each test
- [ ] Remember that `user.click()`, `user.type()`, etc. are async and must be awaited

### Test Organization

- [ ] One test file per source file: `Component.tsx` -> `__tests__/Component.test.tsx`
- [ ] Group related tests with `describe` blocks
- [ ] Use descriptive test names: `it('shows validation error when email is empty')`
- [ ] Follow Arrange-Act-Assert pattern in every test
- [ ] Create a shared `renderWithProviders` helper that wraps components in all necessary providers
- [ ] Keep tests independent -- no test should depend on another test's side effects
- [ ] Clean up after each test (`afterEach` for server.resetHandlers, localStorage.clear, etc.)

## Conventions

### API Design
- **Prefix:** `/api/v1/`
- **Base URL:** `http://localhost:8000` (dev), `https://api.example.com` (prod)
- **Auth Header:** `Authorization: Bearer <JWT_ACCESS_TOKEN>`
- **Pagination:** `offset` (default 0) and `limit` (default 25, max 100)
- **Date Format:** ISO 8601 (YYYY-MM-DD, YYYY-MM-DDTHH:MM:SSZ)
- **Decimal:** JSON decimal as string (e.g., `"amount": "1234.56"`) to avoid precision loss
- **Error Format:**
  ```json
  {
    "status": 400,
    "error": "Bad Request",
    "message": "Validation failed",
    "details": [{"field": "email", "message": "Invalid email format"}],
    "timestamp": "2026-02-24T10:30:00Z",
    "correlation_id": "uuid-here"
  }
  ```

### Authentication & Authorization
- **Token Type:** JWT Bearer
- **Access Token Lifetime:** 15-60 minutes (configurable via `JWT_EXPIRATION` env var)
- **Refresh Token Lifetime:** 7-30 days (optional)
- **Password Requirements:** Minimum 12 characters, complexity check (uppercase, lowercase, numbers, symbols)
- **Roles:** AGENT, UNDERWRITER, CLAIMS_ADJUSTER, FINANCE_MANAGER, RECOVERY_MANAGER, ADMIN, AUDITOR
- **Permission Enforcement:** API-level via @requires_permission decorator, UI-level via useAuth hook

### Logging
- **Format:** Structured JSON (production), human-readable (development)
- **Level:** DEBUG (dev), INFO (prod)
- **Correlation ID:** Included in all logs via X-Correlation-Id header
- **PII Masking:** SSN (XXX-XX-1234), Account (****7890), tokens never logged
- **Output:** stdout (no file logging in production)

### Database Naming
- **Tables:** snake_case (e.g., `policies`, `claim_adjustments`)
- **Columns:** snake_case (e.g., `policy_number`, `created_at`)
- **Primary Keys:** `id` (auto-increment Long)
- **Foreign Keys:** `{table}_id` (e.g., `policy_id`)
- **Timestamps:** `created_at` and `updated_at` on all tables (auto-managed via Base model)
- **Soft Deletes:** Not used (hard delete, audit trail for recovery)

### Code Style
- **Python:** PEP 8 (enforced via `black` formatter, `ruff` linter)
- **Imports:** isort (sorted: standard library, third-party, local)
- **Type Hints:** Required on all function signatures
- **Docstrings:** Google-style on public functions/classes
- **Comments:** Only for non-obvious business logic
- **No Lombok/decorators for boilerplate:** Write explicit code

### Testing
- **Framework:** pytest with pytest-asyncio for FastAPI
- **Fixtures:** Conftest.py for shared fixtures (DB, auth)
- **Mocking:** Use unittest.mock for external service calls
- **Coverage:** Minimum 80% for core business logic
- **Test Data:** Use factory-boy or manual fixtures (no real data)

### Environment Management
- **Dev:** Local `.env` file (git-ignored)
- **Prod:** Environment variables injected at runtime (no .env files)
- **Config:** Pydantic Settings with validation
- **Secrets:** Never in git; use environment variables or secrets manager

### Security Practices
- **Encryption:** AES-256 for PII at rest; HTTPS/TLS 1.2+ in transit
- **Hashing:** bcrypt (strength 10) for passwords
- **Validation:** Input validation at API boundary (Pydantic)
- **SQL Injection:** Prevented by SQLAlchemy ORM (parameterized queries)
- **XSS:** Prevented by JSON-only API (no HTML rendering)
- **CSRF:** Token-based CSRF protection for state-changing operations (if needed)
- **Rate Limiting:** Per-user (1000 req/min), per-endpoint (10k req/min)

### Development Setup
```bash
# Backend setup
cd backend
python -m venv venv
source venv/bin/activate  # or: venv\Scripts\activate (Windows)
```

## Key Architectural Decisions

1. **Layered Architecture**: Clean separation of concerns (Controller → Service → Repository)
2. **Async Processing**: Celery for long-running tasks (payment processing, notifications)
3. **Caching**: Redis for search performance and session data
4. **JWT Auth**: Stateless, scalable authentication suitable for distributed systems
5. **Pydantic for Validation**: Type-safe DTOs with automatic validation and documentation
6. **SQLAlchemy ORM**: Database abstraction, migration support, relationship management
7. **Structured Logging**: Correlation IDs for request tracing across services
8. **Custom Exceptions**: Business logic exceptions with HTTP status mapping
9. **Repository Pattern**: Loose coupling between business logic and data access
