## Policy Lifecycle (State Machine)

```
Quote → Bind → Issue → [Active] → Renew
                         ↓           ↓
                      Endorse     Expire
                         ↓
                      Cancel
```

- **Quote**: Premium estimate, not yet bound
- **Bind**: Policyholder accepts quote, coverage begins on effective date
- **Issue**: Policy document generated, policy number assigned
- **Endorse**: Modify an active policy (add/remove coverage, change limits)
- **Renew**: Extend policy for another term with new premiums
- **Cancel**: Terminate policy (pro-rata or short-rate refund)
- **Expire**: Policy reaches expiration date without renewal

---

## Claim Lifecycle (State Machine)

```
FNOL → Investigate → Adjust → [Settle | Deny] → Close
                       ↓
                   Reopen (if new information)
```

- **FNOL (First Notice of Loss)**: Initial claim filing
- **Investigate**: Gather facts, verify coverage
- **Adjust**: Determine claim amount
- **Settle**: Approve and pay claim
- **Deny**: Reject claim (with reason)
- **Close**: Final disposition
- **Reopen**: New evidence requires re-evaluation

## Key Business Rules

### Underwriting

- Risk classification determines premium tier
- Rating factors: age, location, prior claims, coverage type
- Eligibility criteria: minimum/maximum age, excluded occupations
- Premium = Base Rate x Rating Factor x Coverage Factor - Discounts + Surcharges

### Policy Rules

- Policy number format: `POL-YYYY-NNNNNN` (auto-generated, sequential)
- Effective date must be today or future
- Expiration date must be after effective date
- Coverage limit cannot exceed policy aggregate limit
- Endorsement effective date must be within policy period

### Claim Rules

- Claim number format: `CLM-YYYY-NNNNNN` (auto-generated, sequential)
- Date of loss must be within policy period
- Claimed amount cannot exceed coverage limit
- Claim coverage type must match a coverage on the policy
- Reserve amount adjusts during investigation

### Regulatory

- Grace period for premium payment (varies by state/jurisdiction)
- Cancellation notice requirements (30-day minimum)
- Data retention requirements (7+ years for policy records)
- Annual reporting obligations

## Business Rule Numbering Conventions

Number every rule with domain prefix:

**Policy Rules:** POL-001, POL-002, ...
**Coverage Rules:** COV-001, COV-002, ...
**Premium Rules:** PRM-001, PRM-002, ...
**Claim Rules:** CLM-001, CLM-002, ...

### Example Numbered Rules

**Policy Rules:**
- POL-001: Policy number format: POL-YYYY-NNNNNN (auto-generated)
- POL-002: Effective date >= today for new policies
- POL-003: Term length: 6 months or 12 months
- POL-004: Cancellation requires 30-day notice
- POL-005: Pro-rata refund on cancellation

**Coverage Rules:**
- COV-001: Each coverage has a limit (positive decimal)
- COV-002: Deductible <= limit
- COV-003: Sum of coverage limits <= policy aggregate limit
- COV-004: At least one coverage required per policy

**Premium Rules:**
- PRM-001: Premium = Base Rate x Territory Factor x Age Factor x Claims Factor
- PRM-002: Multi-policy discount: 10% if insured has 2+ active policies
- PRM-003: Claims-free discount: 5% for 3+ years without claims
- PRM-004: Minimum premium: $100 per term

**Claim Rules:**
- CLM-001: Claim number format: CLM-YYYY-NNNNNN
- CLM-002: Date of loss within policy effective/expiration period
- CLM-003: Claimed amount <= coverage limit for the coverage type
- CLM-004: Reserve amount = estimated amount (initial)
- CLM-005: Settlement amount <= reserve amount (unless approved override)

## Business Rules

### Policy Lifecycle

2. Cancellation allowed with pro-rata refund within 30 days
3. Lapse occurs when premium not paid within grace period (15 days)

### Claim Processing
1. FNOL (First Notice of Loss) creates initial claim record
2. Adjuster assigned within 24 hours for claims > $10,000
3. Claims cannot exceed coverage limit
4. Deductible applied before payout calculation

### Premium Calculation
- Base rate × territory factor × vehicle/property factor × driver/age factor × claims history factor
- Discounts: multi-policy (10%), good driver (15%), home security (5%)

## Conventions

### API Design
- **Prefix:** `/api/v1/`
- **Base URL:** `http://localhost:8000` (dev), `https://api.example.com` (prod)
- **Auth Header:** `Authorization: Bearer <JWT_ACCESS_TOKEN>`
- **Pagination:** `offset` (default 0) and `limit` (default 25, max 100)
- **Date Format:** ISO 8601 (YYYY-MM-DD, YYYY-MM-DDTHH:MM:SSZ)
- **Decimal:** JSON decimal as string (e.g., `"amount": "1234.56"`) to avoid precision loss
- **Error Format:**
  ```json
  {
    "status": 400,
    "error": "Bad Request",
    "message": "Validation failed",
    "details": [{"field": "email", "message": "Invalid email format"}],
    "timestamp": "2026-02-24T10:30:00Z",
    "correlation_id": "uuid-here"
  }
  ```

### Authentication & Authorization
- **Token Type:** JWT Bearer
- **Access Token Lifetime:** 15-60 minutes (configurable via `JWT_EXPIRATION` env var)
- **Refresh Token Lifetime:** 7-30 days (optional)
- **Password Requirements:** Minimum 12 characters, complexity check (uppercase, lowercase, numbers, symbols)
- **Roles:** AGENT, UNDERWRITER, CLAIMS_ADJUSTER, FINANCE_MANAGER, RECOVERY_MANAGER, ADMIN, AUDITOR
- **Permission Enforcement:** API-level via @requires_permission decorator, UI-level via useAuth hook

### Logging
- **Format:** Structured JSON (production), human-readable (development)
- **Level:** DEBUG (dev), INFO (prod)
- **Correlation ID:** Included in all logs via X-Correlation-Id header
- **PII Masking:** SSN (XXX-XX-1234), Account (****7890), tokens never logged
- **Output:** stdout (no file logging in production)

### Database Naming
- **Tables:** snake_case (e.g., `policies`, `claim_adjustments`)
- **Columns:** snake_case (e.g., `policy_number`, `created_at`)
- **Primary Keys:** `id` (auto-increment Long)
- **Foreign Keys:** `{table}_id` (e.g., `policy_id`)
- **Timestamps:** `created_at` and `updated_at` on all tables (auto-managed via Base model)
- **Soft Deletes:** Not used (hard delete, audit trail for recovery)

### Code Style
- **Python:** PEP 8 (enforced via `black` formatter, `ruff` linter)
- **Imports:** isort (sorted: standard library, third-party, local)
- **Type Hints:** Required on all function signatures
- **Docstrings:** Google-style on public functions/classes
- **Comments:** Only for non-obvious business logic
- **No Lombok/decorators for boilerplate:** Write explicit code

### Testing
- **Framework:** pytest with pytest-asyncio for FastAPI
- **Fixtures:** Conftest.py for shared fixtures (DB, auth)
- **Mocking:** Use unittest.mock for external service calls
- **Coverage:** Minimum 80% for core business logic
- **Test Data:** Use factory-boy or manual fixtures (no real data)

### Environment Management
- **Dev:** Local `.env` file (git-ignored)
- **Prod:** Environment variables injected at runtime (no .env files)
- **Config:** Pydantic Settings with validation
- **Secrets:** Never in git; use environment variables or secrets manager

### Security Practices
- **Encryption:** AES-256 for PII at rest; HTTPS/TLS 1.2+ in transit
- **Hashing:** bcrypt (strength 10) for passwords
- **Validation:** Input validation at API boundary (Pydantic)
- **SQL Injection:** Prevented by SQLAlchemy ORM (parameterized queries)
- **XSS:** Prevented by JSON-only API (no HTML rendering)
- **CSRF:** Token-based CSRF protection for state-changing operations (if needed)
- **Rate Limiting:** Per-user (1000 req/min), per-endpoint (10k req/min)

## Key Architectural Decisions

1. **Layered Architecture**: Clean separation of concerns (Controller → Service → Repository)
2. **Async Processing**: Celery for long-running tasks (payment processing, notifications)
3. **Caching**: Redis for search performance and session data
4. **JWT Auth**: Stateless, scalable authentication suitable for distributed systems
5. **Pydantic for Validation**: Type-safe DTOs with automatic validation and documentation
6. **SQLAlchemy ORM**: Database abstraction, migration support, relationship management
7. **Structured Logging**: Correlation IDs for request tracing across services
8. **Custom Exceptions**: Business logic exceptions with HTTP status mapping
9. **Repository Pattern**: Loose coupling between business logic and data access
