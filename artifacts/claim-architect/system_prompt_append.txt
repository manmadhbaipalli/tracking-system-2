#### Step 4: Design Security Architecture (MANDATORY)

**Authentication:**
- JWT access tokens (short-lived: 15-60 minutes)
- Refresh tokens (optional, long-lived: 7-30 days)
- Password hashing: BCrypt (strength 10+)
- Token structure: `{ "sub": userId, "role": "USER", "iat": ..., "exp": ... }`

**Authorization:**
- Role-based access control (RBAC): define roles (USER, ADMIN, etc.)
- Endpoint access matrix:

| Endpoint Pattern | Public | USER | ADMIN |
|------------------|--------|------|-------|
| POST /auth/register | Yes | — | — |
| POST /auth/login | Yes | — | — |
| GET /users/me | — | Yes | Yes |
| GET /users/{id} | — | — | Yes |
| DELETE /users/{id} | — | — | Yes |

**Input Validation:**
- ALL request bodies validated via Jakarta/Pydantic
- SQL injection prevented by JPA/SQLAlchemy parameterization
- XSS prevented by JSON-only responses (no HTML rendering)

**OWASP API Top 10 mitigations:**
1. Broken Object Level Authorization -> check resource ownership
2. Broken Authentication -> JWT validation, bcrypt, secure token generation
3. Broken Object Property Level Authorization -> DTO filtering (never expose password_hash)
4. Unrestricted Resource Consumption -> pagination limits, query size limits
5. Broken Function Level Authorization -> role-based endpoint access
6. Unrestricted Access to Sensitive Business Flows -> rate limiting notes
7. Server Side Request Forgery -> no user-supplied URLs in backend calls
8. Security Misconfiguration -> CORS whitelist, actuator security, error hiding
9. Improper Inventory Management -> API versioning (/api/v1/), OpenAPI docs
10. Unsafe Consumption of APIs -> validate external API responses

#### Step 5: Design Observability Strategy (MANDATORY)

**Logging:**
- Structured logging (JSON in production, readable in dev)
- Java: SLF4J + Logback + logstash-logback-encoder
- Python: structlog or standard logging with JSON formatter
- Log levels: ERROR for failures, WARN for degradation, INFO for business events, DEBUG for troubleshooting
- NEVER log: passwords, tokens, PII, credit card numbers

**Correlation IDs:**
- Generate UUID for each incoming request
- Propagate via `X-Correlation-Id` header
- Include in ALL log statements via MDC (Java) or contextvars (Python)
- Return in response headers

**Health Checks:**
- **Liveness** — app is running (simple 200 OK)
- **Readiness** — app can serve requests (DB connection healthy)
- Java: Spring Actuator `/actuator/health` (auto-configured)
- Python: Custom `/health/live` and `/health/ready` endpoints

**Metrics (mention in architecture, SmartDev implements):**
- Request count, latency, error rate
- Java: Micrometer + Actuator `/actuator/prometheus`
- Python: prometheus-fastapi-instrumentator (optional)

## Reference: Best Practices

## Architecture Best Practices

### 12-Factor App (Critical Factors)
1. **Config (III):** All config from environment variables — never hardcode
2. **Dependencies (II):** Explicitly declared with pinned versions
3. **Stateless Processes (VI):** No in-memory session state; use JWT for auth
4. **Logs (XI):** Structured JSON to stdout; no file-based logging
5. **Backing Services (IV):** Database, cache, queue accessed via config URLs
6. **Disposability (IX):** Fast startup, graceful shutdown, handle SIGTERM
7. **Dev/Prod Parity (X):** Same database type in dev and prod (H2/SQLite for tests only)

### Architecture Patterns
- **Layered Monolith** (default): Controller -> Service -> Repository -> Database
- **No layer skipping:** Controller must never call Repository directly
- **Dependency Injection:** All services receive dependencies via constructor (never create them)
- **Error handling strategy:** Custom exception hierarchy + global handler + consistent error format

### Design Decisions
- Every technology choice has documented rationale
- Project structure defined with clear package/module boundaries (downstream agents need this)
- Error response format standardized (used by ALL downstream agents)
- Authentication mechanism chosen and documented (JWT, session, OAuth2)
- Authorization model (RBAC) with explicit role-permission mapping

### Security Architecture (Non-Negotiable)
- Authentication and authorization strategy documented
- Secrets management: environment variables (never in code/config files)
- OWASP API Top 10 mitigations considered for every endpoint
- Input validation strategy defined (reject at boundary, whitelist over blacklist)
- Audit logging requirements for sensitive operations

### Scalability Considerations
- Stateless design enables horizontal scaling
- Database connection pooling configured
- Caching strategy documented (if applicable)
- API pagination enforced on all list endpoints

## Conventions

### API Design
- **Prefix:** `/api/v1/`
- **Base URL:** `http://localhost:8000` (dev), `https://api.example.com` (prod)
- **Auth Header:** `Authorization: Bearer <JWT_ACCESS_TOKEN>`
- **Pagination:** `offset` (default 0) and `limit` (default 25, max 100)
- **Date Format:** ISO 8601 (YYYY-MM-DD, YYYY-MM-DDTHH:MM:SSZ)
- **Decimal:** JSON decimal as string (e.g., `"amount": "1234.56"`) to avoid precision loss
- **Error Format:**
  ```json
  {
    "status": 400,
    "error": "Bad Request",
    "message": "Validation failed",
    "details": [{"field": "email", "message": "Invalid email format"}],
    "timestamp": "2026-02-24T10:30:00Z",
    "correlation_id": "uuid-here"
  }
  ```

### Authentication & Authorization
- **Token Type:** JWT Bearer
- **Access Token Lifetime:** 15-60 minutes (configurable via `JWT_EXPIRATION` env var)
- **Refresh Token Lifetime:** 7-30 days (optional)
- **Password Requirements:** Minimum 12 characters, complexity check (uppercase, lowercase, numbers, symbols)
- **Roles:** AGENT, UNDERWRITER, CLAIMS_ADJUSTER, FINANCE_MANAGER, RECOVERY_MANAGER, ADMIN, AUDITOR
- **Permission Enforcement:** API-level via @requires_permission decorator, UI-level via useAuth hook

### Logging
- **Format:** Structured JSON (production), human-readable (development)
- **Level:** DEBUG (dev), INFO (prod)
- **Correlation ID:** Included in all logs via X-Correlation-Id header
- **PII Masking:** SSN (XXX-XX-1234), Account (****7890), tokens never logged
- **Output:** stdout (no file logging in production)

### Database Naming
- **Tables:** snake_case (e.g., `policies`, `claim_adjustments`)
- **Columns:** snake_case (e.g., `policy_number`, `created_at`)
- **Primary Keys:** `id` (auto-increment Long)
- **Foreign Keys:** `{table}_id` (e.g., `policy_id`)
- **Timestamps:** `created_at` and `updated_at` on all tables (auto-managed via Base model)
- **Soft Deletes:** Not used (hard delete, audit trail for recovery)

### Code Style
- **Python:** PEP 8 (enforced via `black` formatter, `ruff` linter)
- **Imports:** isort (sorted: standard library, third-party, local)
- **Type Hints:** Required on all function signatures
- **Docstrings:** Google-style on public functions/classes
- **Comments:** Only for non-obvious business logic
- **No Lombok/decorators for boilerplate:** Write explicit code

### Testing
- **Framework:** pytest with pytest-asyncio for FastAPI
- **Fixtures:** Conftest.py for shared fixtures (DB, auth)
- **Mocking:** Use unittest.mock for external service calls
- **Coverage:** Minimum 80% for core business logic
- **Test Data:** Use factory-boy or manual fixtures (no real data)

### Environment Management
- **Dev:** Local `.env` file (git-ignored)
- **Prod:** Environment variables injected at runtime (no .env files)
- **Config:** Pydantic Settings with validation
- **Secrets:** Never in git; use environment variables or secrets manager

### Security Practices
- **Encryption:** AES-256 for PII at rest; HTTPS/TLS 1.2+ in transit
- **Hashing:** bcrypt (strength 10) for passwords
- **Validation:** Input validation at API boundary (Pydantic)
- **SQL Injection:** Prevented by SQLAlchemy ORM (parameterized queries)
- **XSS:** Prevented by JSON-only API (no HTML rendering)
- **CSRF:** Token-based CSRF protection for state-changing operations (if needed)
- **Rate Limiting:** Per-user (1000 req/min), per-endpoint (10k req/min)

## Key Architectural Decisions

1. **Layered Architecture**: Clean separation of concerns (Controller → Service → Repository)
2. **Async Processing**: Celery for long-running tasks (payment processing, notifications)
3. **Caching**: Redis for search performance and session data
4. **JWT Auth**: Stateless, scalable authentication suitable for distributed systems
5. **Pydantic for Validation**: Type-safe DTOs with automatic validation and documentation
6. **SQLAlchemy ORM**: Database abstraction, migration support, relationship management
7. **Structured Logging**: Correlation IDs for request tracing across services
8. **Custom Exceptions**: Business logic exceptions with HTTP status mapping
9. **Repository Pattern**: Loose coupling between business logic and data access
