## Reference: Best Practices

## API Design Best Practices

### REST Conventions
- Nouns for resources, HTTP verbs for actions (`POST /orders`, not `POST /createOrder`)
- Consistent plural resource names (`/users`, `/orders`, `/accounts`)
- Standard HTTP status codes: 201 Created, 204 No Content, 400 Bad Request, 404 Not Found, 409 Conflict, 422 Unprocessable Entity
- Use path parameters for resource identity (`/users/{id}`), query parameters for filtering (`/users?status=active`)

### Idempotency
- GET, PUT, DELETE MUST be idempotent (same request = same result)
- POST operations for critical actions (payments, orders) SHOULD accept an `Idempotency-Key` header
- PUT for full replacement, PATCH for partial update — don't mix semantics

### Input Validation
- Every input field validated (type, length, format, range)
- Reject at the boundary — fail fast with field-level error details
- No sensitive data in error messages (no SQL, no stack traces, no internal paths)
- Whitelist allowed characters for string fields where possible

### Pagination
- ALL list endpoints use PageResponse envelope (never return raw arrays)
- Default page size enforced server-side (20)
- Maximum page size enforced server-side (100)
- Never return unbounded result sets

### Error Response Contract
- Consistent error format across ALL endpoints (ErrorResponse with error_code, message, details)
- Field-level validation errors return array of field errors
- Use domain-specific error codes (not just HTTP status codes)

### API Documentation
- OpenAPI spec generated from code annotations (not hand-written)
- Authentication requirements documented per endpoint
- Every endpoint has summary, description, and response status codes
- Request/response examples for complex endpoints

### Insurance-Specific (ACORD)
- Policy number format validation on all policy references (pattern: `POL-YYYYMMDD-XXXXX`)
- Date validation: effective date < expiration date, date of loss within policy period
- Premium amounts: use BigDecimal/Decimal (NEVER float — floating point errors cause premium discrepancies)
- Claim amounts cannot exceed coverage limits (validated at API boundary)
- Policy lifecycle state transitions validated (DRAFT -> ACTIVE -> EXPIRED, no skipping)

```python
from datetime import datetime
from typing import Generic, TypeVar

from pydantic import BaseModel

T = TypeVar("T")


class FieldError(BaseModel):
    field: str
    message: str


class ErrorResponse(BaseModel):
    status: int
    error: str
    message: str
    error_code: str
    details: list[FieldError] | None = None
    timestamp: datetime

    @classmethod
    def create(cls, status: int, error: str, message: str, error_code: str):
        return cls(status=status, error=error, message=message, error_code=error_code, timestamp=datetime.now())


class PageResponse(BaseModel, Generic[T]):
    content: list[T]
    page: int
    size: int
    total_elements: int
    total_pages: int
    first: bool
    last: bool

    @classmethod
    def create(cls, items: list[T], total: int, page: int, size: int):
        total_pages = (total + size - 1) // size if size > 0 else 0
        return cls(
            content=items,
            page=page,
            size=size,
            total_elements=total,
            total_pages=total_pages,
            first=page == 0,
            last=page >= total_pages - 1,
        )
```

```python
from fastapi import APIRouter, Depends, Query, status

from app.schemas.policy import EndorsementRequest, CancelPolicyRequest, PolicyResponse
from app.schemas.common import PageResponse
from app.schemas.enums import PolicyStatus, PolicyType

@router.get("", response_model=PageResponse[PolicyResponse],
            summary="List policies (paginated)")
```

```python
             summary="Bind quote to create a policy",
             responses={400: {"description": "Quote expired or already bound"}})
def bind_quote(quote_id: int, db: Session = Depends(get_db)):
    """Convert a quote into an active policy."""
    return quote_service.bind(db, quote_id)
```

## Conventions

### API Design
- **Prefix:** `/api/v1/`
- **Base URL:** `http://localhost:8000` (dev), `https://api.example.com` (prod)
- **Auth Header:** `Authorization: Bearer <JWT_ACCESS_TOKEN>`
- **Pagination:** `offset` (default 0) and `limit` (default 25, max 100)
- **Date Format:** ISO 8601 (YYYY-MM-DD, YYYY-MM-DDTHH:MM:SSZ)
- **Decimal:** JSON decimal as string (e.g., `"amount": "1234.56"`) to avoid precision loss
- **Error Format:**
  ```json
  {
    "status": 400,
    "error": "Bad Request",
    "message": "Validation failed",
    "details": [{"field": "email", "message": "Invalid email format"}],
    "timestamp": "2026-02-24T10:30:00Z",
    "correlation_id": "uuid-here"
  }
  ```

### Authentication & Authorization
- **Token Type:** JWT Bearer
- **Access Token Lifetime:** 15-60 minutes (configurable via `JWT_EXPIRATION` env var)
- **Refresh Token Lifetime:** 7-30 days (optional)
- **Password Requirements:** Minimum 12 characters, complexity check (uppercase, lowercase, numbers, symbols)
- **Roles:** AGENT, UNDERWRITER, CLAIMS_ADJUSTER, FINANCE_MANAGER, RECOVERY_MANAGER, ADMIN, AUDITOR
- **Permission Enforcement:** API-level via @requires_permission decorator, UI-level via useAuth hook

### Logging
- **Format:** Structured JSON (production), human-readable (development)
- **Level:** DEBUG (dev), INFO (prod)
- **Correlation ID:** Included in all logs via X-Correlation-Id header
- **PII Masking:** SSN (XXX-XX-1234), Account (****7890), tokens never logged
- **Output:** stdout (no file logging in production)

### Database Naming
- **Tables:** snake_case (e.g., `policies`, `claim_adjustments`)
- **Columns:** snake_case (e.g., `policy_number`, `created_at`)
- **Primary Keys:** `id` (auto-increment Long)
- **Foreign Keys:** `{table}_id` (e.g., `policy_id`)
- **Timestamps:** `created_at` and `updated_at` on all tables (auto-managed via Base model)
- **Soft Deletes:** Not used (hard delete, audit trail for recovery)

### Code Style
- **Python:** PEP 8 (enforced via `black` formatter, `ruff` linter)
- **Imports:** isort (sorted: standard library, third-party, local)
- **Type Hints:** Required on all function signatures
- **Docstrings:** Google-style on public functions/classes
- **Comments:** Only for non-obvious business logic
- **No Lombok/decorators for boilerplate:** Write explicit code

### Testing
- **Framework:** pytest with pytest-asyncio for FastAPI
- **Fixtures:** Conftest.py for shared fixtures (DB, auth)
- **Mocking:** Use unittest.mock for external service calls
- **Coverage:** Minimum 80% for core business logic
- **Test Data:** Use factory-boy or manual fixtures (no real data)

### Environment Management
- **Dev:** Local `.env` file (git-ignored)
- **Prod:** Environment variables injected at runtime (no .env files)
- **Config:** Pydantic Settings with validation
- **Secrets:** Never in git; use environment variables or secrets manager

### Security Practices
- **Encryption:** AES-256 for PII at rest; HTTPS/TLS 1.2+ in transit
- **Hashing:** bcrypt (strength 10) for passwords
- **Validation:** Input validation at API boundary (Pydantic)
- **SQL Injection:** Prevented by SQLAlchemy ORM (parameterized queries)
- **XSS:** Prevented by JSON-only API (no HTML rendering)
- **CSRF:** Token-based CSRF protection for state-changing operations (if needed)
- **Rate Limiting:** Per-user (1000 req/min), per-endpoint (10k req/min)

## Key Architectural Decisions

1. **Layered Architecture**: Clean separation of concerns (Controller → Service → Repository)
2. **Async Processing**: Celery for long-running tasks (payment processing, notifications)
3. **Caching**: Redis for search performance and session data
4. **JWT Auth**: Stateless, scalable authentication suitable for distributed systems
5. **Pydantic for Validation**: Type-safe DTOs with automatic validation and documentation
6. **SQLAlchemy ORM**: Database abstraction, migration support, relationship management
7. **Structured Logging**: Correlation IDs for request tracing across services
8. **Custom Exceptions**: Business logic exceptions with HTTP status mapping
9. **Repository Pattern**: Loose coupling between business logic and data access
