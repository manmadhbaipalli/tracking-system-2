#### Step 4: Design Security Architecture (MANDATORY)

Define all security concerns:

**Authentication:**
- JWT access tokens (short-lived: 15-60 minutes)
- Refresh tokens (optional, long-lived: 7-30 days)
- Password hashing: BCrypt (strength 10+)
- Token structure: `{ "sub": userId, "role": "USER", "iat": ..., "exp": ... }`

**Authorization:**
- Role-based access control (RBAC): define roles (USER, ADMIN, etc.)
- Endpoint access matrix:

| Endpoint Pattern | Public | USER | ADMIN |
|------------------|--------|------|-------|
| POST /auth/register | Yes | — | — |
| POST /auth/login | Yes | — | — |
| GET /users/me | — | Yes | Yes |
| GET /users/{id} | — | — | Yes |
| DELETE /users/{id} | — | — | Yes |

**Input Validation:**
- ALL request bodies validated via Jakarta/Pydantic
- SQL injection prevented by JPA/SQLAlchemy parameterization
- XSS prevented by JSON-only responses (no HTML rendering)

**OWASP API Top 10 mitigations:**
1. Broken Object Level Authorization → check resource ownership
2. Broken Authentication → JWT validation, bcrypt, secure token generation
3. Broken Object Property Level Authorization → DTO filtering (never expose password_hash)
4. Unrestricted Resource Consumption → pagination limits, query size limits
5. Broken Function Level Authorization → role-based endpoint access
6. Unrestricted Access to Sensitive Business Flows → rate limiting notes
7. Server Side Request Forgery → no user-supplied URLs in backend calls
8. Security Misconfiguration → CORS whitelist, actuator security, error hiding
9. Improper Inventory Management → API versioning (/api/v1/), OpenAPI docs
10. Unsafe Consumption of APIs → validate external API responses

#### Step 5: Design Observability Strategy (MANDATORY)

**Logging:**
- Structured logging (JSON in production, readable in dev)
- Java: SLF4J + Logback + logstash-logback-encoder
- Python: structlog or standard logging with JSON formatter
- Log levels: ERROR for failures, WARN for degradation, INFO for business events, DEBUG for troubleshooting
- NEVER log: passwords, tokens, PII, credit card numbers

**Correlation IDs:**
- Generate UUID for each incoming request
- Propagate via `X-Correlation-Id` header
- Include in ALL log statements via MDC (Java) or contextvars (Python)
- Return in response headers

**Health Checks:**
- **Liveness** — app is running (simple 200 OK)
- **Readiness** — app can serve requests (DB connection healthy)
- Java: Spring Actuator `/actuator/health` (auto-configured)
- Python: Custom `/health/live` and `/health/ready` endpoints

**Metrics (mention in architecture, SmartDev implements):**
- Request count, latency, error rate
- Java: Micrometer + Actuator `/actuator/prometheus`
- Python: prometheus-fastapi-instrumentator (optional)

#### Step 6: Design Configuration Strategy (MANDATORY)

Follow 12-Factor App principles:
- **Config via environment variables** — never hardcode secrets
- **Profiles** — dev (local, H2/SQLite), prod (PostgreSQL, secure defaults)
- **Defaults** — safe defaults for all config values
- **Secrets** — JWT secret, DB password, API keys via env vars only

**Environment variables to define:**

| Variable | Description | Default |
|----------|-------------|---------|
| DATABASE_URL | Database connection string | sqlite:///./dev.db or jdbc:h2:mem:devdb |
| JWT_SECRET | JWT signing key | (MUST be set in prod, no default) |
| JWT_EXPIRATION | Token lifetime in seconds | 3600 |
| CORS_ORIGINS | Allowed origins | http://localhost:3000 |
| LOG_LEVEL | Logging level | INFO |
| SERVER_PORT | Application port | 8080 / 8000 |

## Key Patterns
- **Dependency Injection**: Use FastAPI's Depends() for database sessions, auth
- **Repository Pattern**: Services layer abstracts database operations with async methods
- **Pydantic Models**: Separate request/response schemas from database models with validation
- **Async Context**: All database operations use async/await patterns with proper session management
- **Audit Trail**: Every data modification includes user_id and timestamp with comprehensive tracking
- **Security**: Sensitive data (SSN/TIN, payments) masked in responses and encrypted at rest
- **Error Responses**: Consistent error format with HTTPException and specific messages per requirements
- **Role-Based Access**: Decorator-based route protection by user role (admin, agent, adjuster)
- **Transaction Management**: Payment operations wrapped in database transactions
- **External Integrations**: Circuit breaker pattern with retry logic for resilience

## Data Architecture Patterns
- **Policy-Claim Relationships**: Foreign key relationships with claim-level policy overrides
- **Payment Complex Structures**: Multiple payees, reserve allocations, void/reversal chains
- **Audit Polymorphism**: Audit trails for all entity types with proper change tracking
- **Search Optimization**: Composite database indexes for performance-critical searches
- **Data Masking**: Consistent masking of PII across all API responses and logs

## Performance Requirements
- **Policy Search**: Results within 3 seconds (9+ search criteria support)
- **Policy/Claim Details**: Load within 5 seconds with full relationship data
- **Payment Processing**: Complete within 5 seconds including external validations
- **Concurrent Sessions**: Support multiple users without performance degradation
- **Database Indexes**: Optimized for search patterns and relationship queries
