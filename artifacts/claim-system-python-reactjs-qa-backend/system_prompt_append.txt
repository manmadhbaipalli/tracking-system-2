## Reference: Best Practices

## Testing Best Practices

### Test Pyramid
- Many unit tests (fast, isolated, mock dependencies with unittest.mock)
- Fewer integration tests (TestClient with real SQLite DB, full HTTP stack)
- At least 1 end-to-end test per critical business flow

### Test Naming Convention
- Pattern: `test_[action]_[scenario]_[expected_result]`
- Example: `test_create_user_with_duplicate_email_returns_409`
- BAD: `test_1`, `test_user`, `test_create`

### Test Structure (Arrange-Act-Assert)
- **Arrange:** Set up test data, configure fixtures and mocks
- **Act:** Call the function/endpoint under test
- **Assert:** Verify the expected outcome with specific assertions
- Each test tests ONE thing — one logical assertion per test
- Use descriptive test function names (they ARE the documentation)

### Test Isolation
- Unit tests: mock DB session with `unittest.mock.patch` or `AsyncMock`
- API tests: use `httpx.AsyncClient` with dependency overrides for DB
- Integration tests: use SQLite in-memory DB (override `get_db` dependency)
- NEVER connect to external databases in tests
- Use `pytest.fixture` for shared setup (client, DB session, test data)

### Coverage Targets
- Service layer: >= 80% line coverage
- Router layer: >= 70% (happy path + error paths)
- Do NOT waste effort testing schemas, config classes, or simple models
- Focus on business logic and error handling paths

### Security Tests (MANDATORY)
- Unauthenticated access to protected endpoints returns 401
- Wrong role accessing admin endpoints returns 403
- User A cannot access User B's resources (object-level auth)
- Passwords never appear in any API response body
- Input validation: oversized input, special characters, SQL injection strings

## Conventions

### API Design
- **Prefix:** `/api/v1/`
- **Base URL:** `http://localhost:8000` (dev), `https://api.example.com` (prod)
- **Auth Header:** `Authorization: Bearer <JWT_ACCESS_TOKEN>`
- **Pagination:** `offset` (default 0) and `limit` (default 25, max 100)
- **Date Format:** ISO 8601 (YYYY-MM-DD, YYYY-MM-DDTHH:MM:SSZ)
- **Decimal:** JSON decimal as string (e.g., `"amount": "1234.56"`) to avoid precision loss
- **Error Format:**
  ```json
  {
    "status": 400,
    "error": "Bad Request",
    "message": "Validation failed",
    "details": [{"field": "email", "message": "Invalid email format"}],
    "timestamp": "2026-02-24T10:30:00Z",
    "correlation_id": "uuid-here"
  }
  ```

### Authentication & Authorization
- **Token Type:** JWT Bearer
- **Access Token Lifetime:** 15-60 minutes (configurable via `JWT_EXPIRATION` env var)
- **Refresh Token Lifetime:** 7-30 days (optional)
- **Password Requirements:** Minimum 12 characters, complexity check (uppercase, lowercase, numbers, symbols)
- **Roles:** AGENT, UNDERWRITER, CLAIMS_ADJUSTER, FINANCE_MANAGER, RECOVERY_MANAGER, ADMIN, AUDITOR
- **Permission Enforcement:** API-level via @requires_permission decorator, UI-level via useAuth hook

### Logging
- **Format:** Structured JSON (production), human-readable (development)
- **Level:** DEBUG (dev), INFO (prod)
- **Correlation ID:** Included in all logs via X-Correlation-Id header
- **PII Masking:** SSN (XXX-XX-1234), Account (****7890), tokens never logged
- **Output:** stdout (no file logging in production)

### Database Naming
- **Tables:** snake_case (e.g., `policies`, `claim_adjustments`)
- **Columns:** snake_case (e.g., `policy_number`, `created_at`)
- **Primary Keys:** `id` (auto-increment Long)
- **Foreign Keys:** `{table}_id` (e.g., `policy_id`)
- **Timestamps:** `created_at` and `updated_at` on all tables (auto-managed via Base model)
- **Soft Deletes:** Not used (hard delete, audit trail for recovery)

### Code Style
- **Python:** PEP 8 (enforced via `black` formatter, `ruff` linter)
- **Imports:** isort (sorted: standard library, third-party, local)
- **Type Hints:** Required on all function signatures
- **Docstrings:** Google-style on public functions/classes
- **Comments:** Only for non-obvious business logic
- **No Lombok/decorators for boilerplate:** Write explicit code

### Testing
- **Framework:** pytest with pytest-asyncio for FastAPI
- **Fixtures:** Conftest.py for shared fixtures (DB, auth)
- **Mocking:** Use unittest.mock for external service calls
- **Coverage:** Minimum 80% for core business logic
- **Test Data:** Use factory-boy or manual fixtures (no real data)

### Environment Management
- **Dev:** Local `.env` file (git-ignored)
- **Prod:** Environment variables injected at runtime (no .env files)
- **Config:** Pydantic Settings with validation
- **Secrets:** Never in git; use environment variables or secrets manager

### Security Practices
- **Encryption:** AES-256 for PII at rest; HTTPS/TLS 1.2+ in transit
- **Hashing:** bcrypt (strength 10) for passwords
- **Validation:** Input validation at API boundary (Pydantic)
- **SQL Injection:** Prevented by SQLAlchemy ORM (parameterized queries)
- **XSS:** Prevented by JSON-only API (no HTML rendering)
- **CSRF:** Token-based CSRF protection for state-changing operations (if needed)
- **Rate Limiting:** Per-user (1000 req/min), per-endpoint (10k req/min)

## Commands

### Development Setup
```bash
# Backend setup
cd backend
python -m venv venv
source venv/bin/activate  # or: venv\Scripts\activate (Windows)
pip install -r requirements-dev.txt
python -m alembic upgrade head  # Run DB migrations

cd ../frontend
npm install
npm run dev  # Start dev server (http://localhost:3000)
```

```bash

cd backend
docker build -t claim-system:latest .

# Backend: Run tests locally
pytest tests/ -v --cov=app --cov-report=html

npm run build  # Output to build/ directory
```

```bash

pytest tests/ -v

pytest tests/ --cov=app --cov-report=term-missing
```

```bash

python -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

gunicorn app.main:app -w 4 -b 0.0.0.0:8000

docker-compose up -d

# Frontend: Development
npm run dev

npm start
```



## Key Architectural Decisions

1. **Layered Architecture**: Clean separation of concerns (Controller → Service → Repository)
2. **Async Processing**: Celery for long-running tasks (payment processing, notifications)
3. **Caching**: Redis for search performance and session data
4. **JWT Auth**: Stateless, scalable authentication suitable for distributed systems
5. **Pydantic for Validation**: Type-safe DTOs with automatic validation and documentation
6. **SQLAlchemy ORM**: Database abstraction, migration support, relationship management
7. **Structured Logging**: Correlation IDs for request tracing across services
8. **Custom Exceptions**: Business logic exceptions with HTTP status mapping
9. **Repository Pattern**: Loose coupling between business logic and data access
